# 03 - 编写技巧与最佳实践

## 3.1 Description 黄金公式

```
[做什么] - [一句话价值]。当用户说"关键词1"、"关键词2"、"关键词3"时自动激活。
```

或英文：
```
Use when [scenario1], [scenario2]. Covers [capability1], [capability2].
```

### 关键词覆盖策略

至少覆盖以下三类词：
1. **动词**：写、生成、检查、修复、审核、部署……
2. **名词/产物**：API、ADR、需求、测试、Bug、PR……
3. **同义词/口语**：bug / 报错 / 出错；deploy / 部署 / 上线

✅ 实战示例（来自 `bug-fix`）：
```
Bug分析修复助手 - 深度分析Bug根因，定位代码，给出修复方案。
当用户说"修复bug"、"排查问题"、"报错了"、"fix"时自动激活。
```

## 3.2 渐进加载实践

### 原则：SKILL.md 不超过 500 行

超出后按以下方式拆分：

```
my-skill/
├── SKILL.md                # 决策路径 + 工作流总览
├── references/
│   ├── templates.md        # 长模板放这里
│   ├── checklist.md        # 详细 checklist
│   └── examples.md         # 完整示例
└── scripts/
    └── helper.sh
```

`SKILL.md` 中只写：
```markdown
## 模板
详见 [templates.md](./references/templates.md)。

## 检查清单
执行前对照 [checklist.md](./references/checklist.md)。
```

### 反例：单文件巨型 Skill
- 4000 行 SKILL.md → Agent 加载就吃掉一半上下文 → 后续推理质量下降。

## 3.3 工作流写作技巧

### ① 用编号步骤，不用散文

❌：
> 首先你需要确定需求，然后想清楚要做什么，接着开始写……

✅：
```markdown
### 第一步：确定 ADR 编号
ls docs/adr/ | sort -n | tail -1

### 第二步：收集信息
向用户询问以下 4 个问题…

### 第三步：生成文件
使用 [模板](./assets/adr-template.md) 创建 docs/adr/{编号}-{标题}.md
```

### ② 给出"决策点"而非"建议"

让 Agent 知道**何时分支**：

```markdown
如果 `docs/adr/` 不存在 → 从 0001 开始
否则 → 在最大编号 +1
```

### ③ 输出格式必须明确

如果 Skill 有固定产物（如评分报告、ADR 文档），**直接给出 Markdown 模板**，让 Agent 复制填空。

## 3.4 决策边界（最容易混淆的三组）

### Skill vs Instruction
| | Skill | Instruction |
|---|---|---|
| 适用范围 | **特定任务** | **大多数工作** |
| 加载时机 | 用户提及触发词时 | 文件匹配 applyTo 或描述命中 |
| 例子 | "写ADR"、"生成测试" | "所有 Python 文件用 typing" |

判断：**这条规则适用于"大部分工作"还是"具体某类任务"？**

### Skill vs Prompt
| | Skill | Prompt |
|---|---|---|
| 多步骤 | 是 | 通常 1-2 步 |
| 捆绑资产 | 支持 scripts/templates | 不支持 |
| 参数化输入 | 弱 | 强（`$ARGUMENTS` 是主角） |

判断：**需要随附脚本/模板/参考文档吗？需要 → Skill。**

### Skill vs Agent
| | Skill | Custom Agent |
|---|---|---|
| 上下文 | 与主 agent 共享 | **隔离**（subagent 返回单一结果） |
| 工具限制 | 全工作流统一 | 每阶段不同 |
| 角色 | 工作流 | 角色 + 工具集 |

判断：**需要上下文隔离或多阶段不同工具集吗？需要 → Agent。**

## 3.5 反模式清单（直接抄走避坑）

| 反模式 | 后果 | 修复 |
|--------|------|------|
| 描述太泛（"一个有用的工具"） | 永不被发现 | 加触发关键词 |
| name 与文件夹名不一致 | 静默失败 | 改成一致 |
| 单 SKILL.md 4000 行 | 吃光上下文 | 拆 references/ |
| 资源引用嵌套两层以上 | Agent 不会跟进 | 扁平化到一层 |
| 用绝对路径 `/Users/xxx/...` | 换机器就坏 | 用 `./` 相对路径 |
| 多个关注点塞一个 Skill | 触发混乱 | 一个 Skill 一个职责 |
| Frontmatter 描述含未转义冒号 | YAML 解析失败 | 加引号 |
| 工作流只写"建议你这样做" | Agent 不知怎么执行 | 写明步骤+决策点 |

## 3.6 自检清单（提交前过一遍）

- [ ] `name` = 文件夹名（小写、连字符）
- [ ] `description` 含 ≥3 个触发关键词
- [ ] `description` 写明"做什么"+"何时用"
- [ ] SKILL.md 行数 < 500
- [ ] 所有资源引用使用 `./` 相对路径
- [ ] 引用深度只有一层（直接子目录）
- [ ] 工作流是编号步骤，不是散文
- [ ] 有明确的输出格式（如适用）
- [ ] 在 VS Code 输入 `/` 能看到此 Skill（验证 frontmatter 正确）
- [ ] 测试用真实触发词，能被自动加载
