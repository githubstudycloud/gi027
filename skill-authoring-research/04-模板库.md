# 04 - 模板库

按 5 种典型场景给出可直接复制修改的 Skill 模板。

---

## 模板 A：工作流型 Skill（最常用）

适合：ADR、Bug 修复、需求分析等多步骤流程。

```markdown
---
name: my-workflow
description: 简介 - 价值描述。当用户说"关键词1"、"关键词2"、"关键词3"时自动激活。
argument-hint: "[简要参数]"
---

# 我的工作流

## 何时使用
- 场景 1
- 场景 2

## 参数说明
- `$ARGUMENTS`：xxx

## 工作流程

### 第一步：收集信息
向用户询问：
1. 问题 A？
2. 问题 B？

### 第二步：执行操作
（具体命令/检查项）

### 第三步：生成产物
使用 [模板](./assets/output-template.md) 创建文件。

## 输出格式
```markdown
（输出模板）
```
```

## 模板 B：工具型 Skill（连接外部系统）

适合：数据库管理、消息队列、API 调用。

```markdown
---
name: xxx-manager
description: XXX 管理助手，帮助用户连接和操作 XXX。当用户想要查询 XXX、操作 XXX、管理 XXX 时自动激活。
argument-hint: "[操作类型] [参数]"
---

# XXX 管理助手

## 支持的操作
| 命令 | 说明 |
|------|------|
| list | 列出资源 |
| get | 查询详情 |
| create | 创建资源 |

## 配置
- 配置文件位置：`~/.config/xxx.yml`
- 必需字段：`host`、`port`、`token`

## 工作流程
### 第一步：解析操作意图
### 第二步：检查连接配置
### 第三步：执行命令
（用 [helper 脚本](./scripts/helper.sh)）

### 第四步：格式化输出
```

## 模板 C：写作型 Skill

适合：技术文档、营销文案、学术论文等内容生成。

```markdown
---
name: write-xxx
description: XXX 写作助手 - 帮助撰写 XXX 类内容。当用户说"写XXX"、"帮我写XXX"时自动激活。
argument-hint: "[主题或要点]"
---

# XXX 写作助手

## 何时使用
- 场景 1
- 场景 2

## 写作流程

### 第一步：明确目标
- 受众是谁？
- 核心信息是什么？
- 字数要求？

### 第二步：选择结构
参考 [结构模板](./references/structures.md)。

### 第三步：撰写正文
遵循 [风格指南](./references/style-guide.md)。

### 第四步：自检
对照 [检查清单](./references/checklist.md)。
```

## 模板 D：编排型 Skill（调度子专家）

适合：审核类工作流，根据输入路由到不同 subagent。

```markdown
---
name: orchestrator
description: 智能编排器 - 自动识别任务类型并调度子专家。当用户说"审核"、"分析"时自动激活。
argument-hint: "[内容] 或 [类型] [内容]"
allowed-tools: Read, Write, Agent
---

# 编排器

## 第一步：识别类型
| 代码 | 特征 |
|------|------|
| typeA | 关键词 A |
| typeB | 关键词 B |

如无法判断 → 询问用户。

## 第二步：确认范围
向用户简述识别结果，等待"继续"。

## 第三步：调度子专家
| 类型 | Agent |
|------|-------|
| typeA | `xxx-typeA-expert` |
| typeB | `xxx-typeB-expert` |

## 第四步：整合输出
按 [报告模板](./assets/report-template.md) 汇总。
```

## 模板 E：规范检查型 Skill

适合：代码规范、安全审查。

```markdown
---
name: check-xxx
description: XXX 规范检查 - 对照标准检查代码/配置。当用户说"检查xxx"、"审查xxx"时自动激活。
---

# XXX 规范检查

## 检查维度
1. 维度 A
2. 维度 B
3. 维度 C

## 工作流程
### 第一步：确定检查范围
（让用户指定文件/目录，或全项目）

### 第二步：逐项检查
对照 [规则清单](./references/rules.md) 逐项扫描。

### 第三步：生成报告
```markdown
| 规则 | 状态 | 文件:行 | 修复建议 |
|------|------|---------|---------|
| R001 | ❌   | foo.ts:42 | 改为 xxx |
```
```
