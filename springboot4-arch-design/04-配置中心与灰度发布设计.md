# 04 - 配置中心与灰度发布设计

> 版本：v1.0 | 日期：2026-04-27 | 状态：草稿

---

## 1. 配置中心

### 1.1 选型

| 选项 | 优点 | 缺点 | 选用 |
|------|------|------|------|
| Nacos 2.x | 配置 + 注册一体、国内生态 | 多租户能力一般 | ✅ 主选 |
| Apollo | 灰度发布、权限模型成熟 | 部署组件多 | ⚠️ 备选 |
| Spring Cloud Config + Git | 与 GitLab 强一致、可审计 | 推送延迟、需 Bus | ✅ 兜底（Git 作为 SoR） |

**最终方案**：Nacos 作为运行时配置源 + GitLab 作为 SoR（Source of Record）。所有变更**先提 MR 到 GitLab**，合并后由 CI 同步到 Nacos；保留人工应急通道（直改 Nacos，但事后必须回写 GitLab）。

### 1.2 配置分层

```
1. 代码默认值（@ConfigurationProperties + record 默认）
2. application-{profile}.yml（构建期，仅与"代码强相关"项）
3. Nacos：dataId=app-name, group={env}                ← 应用级
4. Nacos：dataId=common-{env}, group=COMMON           ← 公共（DB、MQ、Redis...）
5. Nacos：dataId=app-name-{instanceId}, group={env}   ← 实例级（灰度）
6. 启动参数 -D / 环境变量                              ← 紧急覆盖
```

下层覆盖上层。

### 1.3 配置变更总线（呼应你的需求）

```
开发者 ──► GitLab MR ──► 评审合并 ──► CI/CD
                                       │
                                       ▼
                                ┌─────────────┐
                                │  Nacos      │  写入 + 版本号
                                └──────┬──────┘
                                       │ long-poll / push
                          ┌────────────┼────────────┐
                          ▼            ▼            ▼
                     [实例 1]      [实例 2]      [实例 N]
                       │              │              │
                       ▼              ▼              ▼
                 ConfigChangeBus（应用内事件总线）
                       │
                       ▼
              监听器（DataSourceRegistry / FeatureFlag / Routing / ...）
```

- 配置变更 = **GitLab 提交记录** + **Nacos 版本** + **应用内事件**，三处可追溯；
- 关键配置（DB / 路由 / 鉴权策略）必须双人审批。

### 1.4 不重启变更（Hot Reload）

| 类别 | 是否支持热更 | 实现 |
|------|------------|------|
| 业务参数（阈值、开关） | ✅ | `@RefreshScope` + ConfigChangeBus |
| 数据源新增/移除 | ✅ | DataSourceRegistry（文档 02） |
| 路由 / 限流规则 | ✅ | Sentinel + 推送 |
| 日志级别 | ✅ | Spring Boot Actuator `loggers` |
| 线程池参数 | ✅ | DynamicTp |
| 加密密钥 | ⚠️ 半热更 | 双密钥滚动 |
| 框架级（Spring Boot 自身） | ❌ | 走滚动重启 |

## 2. 网关与接口稳定（呼应"保留地址换后端服务"）

### 2.1 网关定位

```
[客户端固定地址]  api.example.com/v1/order
          │
          ▼
   ┌──────────────┐    动态路由（Nacos 推送）
   │   Gateway    │ ◄────────────────────────
   └──────┬───────┘
          │
   ┌──────┼─────────┬───────────────┐
   ▼      ▼         ▼               ▼
 [order-svc-v1] [order-svc-v2] [order-svc-v3-canary]
```

- **对外 URL 永不变**；内部服务名 / 版本 / 实例 / 灰度策略由网关动态决定。
- 路由规则存 **GitLab + Nacos**，通过总线推送，**无需 nginx reload**。

### 2.2 接口版本策略

- URL 加版本号 `/v1/...`，新版与旧版**并行 6 个月**，期满下线；
- 网关支持基于 Header / 用户灰度位 / 比例 路由到不同版本；
- 接口契约（OpenAPI）入 GitLab，CI 强制兼容性校验。

## 3. 灰度与缓慢变更（无需停止服务）

### 3.1 流量灰度（应用层）

| 维度 | 实现 |
|------|------|
| 按比例 | 网关权重路由 |
| 按用户标签 | 网关读取 JWT claim → 分流 |
| 按地域 / 渠道 | Header 路由 |
| 按租户 | tenantId 哈希分桶 |

### 3.2 实例滚动（部署层）

```
旧实例池：A1 A2 A3 A4 A5
新实例池：               B1 B2 B3 B4 B5

阶段 1：B1 上线 → 健康检查 → 流量 5%
阶段 2：B1+B2 上线 → 流量 25%
阶段 3：B1..B3 → 流量 50%，A1..A2 进入 draining
阶段 4：全量切换 B
阶段 5：A 全部下线
任一阶段告警 → 自动回退
```

- **Draining**：实例从注册中心摘除 + 等待飞行中请求结束 + 关闭，业务无感知；
- 所有阶段由 K8s（生产）/ Swarm（beta）控制，对应文档 07。

### 3.3 数据库变更（最难）

- 强制 **expand-contract** 模式：
  1. expand：加列/加表（兼容老代码）；
  2. 双写 / 灰度读新；
  3. 切流；
  4. contract：删旧列/旧表（独立窗口、独立 MR）；
- 禁止"一次性 alter + 改代码 + 上线"。

## 4. 配置安全

- 敏感配置（密码、密钥、Token）必须 KMS 加密存储；启动时解密；Nacos 控制台不可见明文；
- 审计：每次配置变更落 `t_config_change_log`（who / when / what / why / MR 链接）；
- 回退：保留最近 30 个版本，一键回退。
