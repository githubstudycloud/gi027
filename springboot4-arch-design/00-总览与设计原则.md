## 2. 设计目标

1. **多项目可拆可合**：所有横切能力以 starter 形式提供，业务项目按需引入，不强绑。
2. **可演进**：技术选型支持"双轨并行 + 缓慢替换"，避免一次性大爆炸升级。
3. **可观测可定位**：异常、日志、链路、配置变更全链路可追溯。
4. **可回退**：所有变更（配置、版本、流量）必须可在 5 分钟内回退。
5. **规范前置**：通过 starter / archetype / 编译期检查把规范变成"默认行为"，而非靠人自觉。

## 3. 核心设计原则（横切，全模块强制）

### 3.1 异常处理原则

| 层级 | 是否允许 `catch (Exception e)` | 说明 |
|------|--------------------------------|------|
| Controller / Facade | ❌ 禁止 | 由全局 `@RestControllerAdvice` 统一托底 |
| Service（业务方法） | ❌ 禁止 | 只允许捕获已知受检异常并转换 |
| Service（**编排方法**，标注 `@PartialFailureTolerant`）| ⚠️ 允许，但必须落库 + 上报 | 用于"批量 / 多步骤允许部分失败"的场景 |
| Adapter / Infrastructure（外部 IO） | ✅ 允许 | 必须立即转换为领域异常 `DomainException` 或 `IntegrationException` |
| 全局兜底 `GlobalExceptionHandler` | ✅ 必须 | 所有未识别异常 → `5xx`，并落 `error_event` 表 |

**统一异常基类（草案）：**

```
BaseException
 ├─ BusinessException       // 业务规则不通过，4xx
 ├─ DomainException         // 领域不变量被破坏，5xx，告警
 ├─ IntegrationException    // 第三方 / DB / MQ 失败，5xx，可重试
 └─ PartialFailureException // 编排方法收集子失败，2xx + body 标记
```

**部分失败模型**（关键约定）：

- 同一编译单元内（同一子项目内）：**不允许**用接口调用绕过事务，必须显式声明 `@PartialFailureTolerant`，并通过 `PartialResult<T>` 收集每一项结果。
- 跨子项目：通过**接口契约 + 独立事务**调用，天然部分失败安全。

### 3.2 复杂度与代码块规范

| 指标 | 阈值 | 检查方式 |
|------|------|---------|
| 圈复杂度（Cyclomatic Complexity） | ≤ 4 | Checkstyle + SonarQube，CI 强制 |
| 单方法行数 | ≤ 50 行 | Checkstyle，CI 强制 |
| 单文件行数 | ≤ 500 行 | Checkstyle，警告级 |
| 方法参数 | ≤ 5 | 超出强制改为参数对象 |
| 嵌套深度 | ≤ 3 | Checkstyle |

> 落地：在 `enterprise-build-bom` 中提供统一 `checkstyle.xml` / `pmd.xml` / `spotbugs-exclude.xml`，所有子模块继承。

### 3.3 模块通信规范

| 关系 | 通信方式 |
|------|---------|
| 同一子项目内（同 JVM、同 deploy unit） | **直接方法调用**；不允许为了"解耦"而 HTTP 自调；可定义内部 `Port`（六边形端口）做面向接口编程 |
| 同进程跨子模块 | 通过 `*-api` 模块发布 SPI / Facade 接口；实现模块通过 Spring Bean 注入 |
| 跨子项目（跨进程） | 必须走 `*-client` 模块（Feign / OpenFeign / RestClient 封装），并强制熔断 + 超时 |

### 3.4 包年龄策略（4 年窗口）

- 所有 Maven 依赖**首发版本必须在最近 4 年内**（CI 通过 `versions-maven-plugin` + 自定义脚本校验）。
- 超期但暂无替代品 → 移入 `enterprise-legacy-lib/`，以 `system` scope + `<systemPath>` 引入；流水线只校验从仓库拉取的部分，本地包豁免。
- 详见文档 06。

### 3.5 工具类封装

- 所有第三方工具类（Hutool / Guava / Apache Commons / Jackson 等）**不允许**业务代码直接 import；必须通过 `enterprise-common-utils` 中的封装类调用。
- 目的：将来替换底层实现（例如 Hutool → 自研 / Guava）只改一处。

### 3.6 配置规范

- 三层：**默认值（代码）→ 环境配置（Nacos）→ 实例覆盖（启动参数）**。
- 禁止在业务代码中 `@Value` 直接读取；必须通过 `@ConfigurationProperties` + 不可变记录类（Java 21 record）。
- 详见文档 04。
