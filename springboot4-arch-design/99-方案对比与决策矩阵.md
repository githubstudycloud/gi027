# 99 - 方案对比与决策矩阵

> 版本：v1.0 | 日期：2026-04-27 | 状态：草稿
> 用途：对 00–07 文档中**有明确分歧的关键选型**做横向对比，输出推荐 + 风险

---

## 1. 模块组织：六边形 vs 传统三层 vs DDD 战术

| 维度 | 六边形（Ports & Adapters） | 传统三层（Controller/Service/DAO） | DDD 战术（Aggregate/Repository） |
|------|---------------------------|-----------------------------------|-------------------------------|
| 学习成本 | 中 | 低 | 高 |
| 可测试性 | ★★★★★ | ★★ | ★★★★ |
| 业务复杂度上限 | 高 | 中 | 极高 |
| 与 Spring 耦合 | 内层 0 耦合 | 强耦合 | 内层 0 耦合 |
| 适合多项目 | ✅ | ⚠️ | ⚠️（重） |
| 推荐 | **✅ 默认** | 适合 CRUD 项目 | 复杂域单独引入 |

**决策**：默认六边形，archetype 提供两种模板（六边形 / 三层）。

## 2. 配置中心：Nacos vs Apollo vs Spring Cloud Config

| 维度 | Nacos | Apollo | SCC + Git |
|------|-------|--------|-----------|
| 配置 + 注册一体 | ✅ | ❌ | ❌ |
| 灰度发布 | 中 | ★★★★ | 弱 |
| 权限模型 | 中 | ★★★★ | 依赖 Git |
| 多语言 SDK | ✅ | ✅ | 弱 |
| 国内生态 | ★★★★★ | ★★★★ | ★★ |
| 与 GitLab 协同 | 需自建 | 需自建 | ★★★★★（天然） |
| 推荐 | **✅ 主选** | 强治理场景备选 | 兜底 SoR |

**决策**：Nacos 作运行时，GitLab 作 SoR，二者通过 CI 同步。

---

## 3. 多数据源：Spring 原生 vs dynamic-datasource vs ShardingSphere

| 维度 | Spring `AbstractRoutingDataSource` | dynamic-datasource (Baomidou) | ShardingSphere-JDBC |
|------|---------------------------------|------------------------------|---------------------|
| 动态加减库 | 需自实现 | ✅ | ✅ |
| 注解切换 | 需自实现 | ✅ `@DS` | 配置驱动 |
| 分库分表 | ❌ | ❌ | ✅ |
| 读写分离 | 需自实现 | ✅ | ✅ |
| 学习成本 | 低 | 低 | 中 |
| 与 MyBatis-Plus 兼容 | 好 | ★★★★★ | 好 |
| 推荐 | 简单场景 | **✅ 默认（包公共库 + 业务库路由）** | 业务库需分片时叠加 |

**决策**：`dynamic-datasource` 作为默认；ShardingSphere 作为可选模块。

## 4. MQ：RocketMQ vs Kafka vs RabbitMQ vs Pulsar

| 维度 | RocketMQ | Kafka | RabbitMQ | Pulsar |
|------|----------|-------|----------|--------|
| 业务消息 | ★★★★★ | ★★★ | ★★★★ | ★★★★ |
| 流式 / 高吞吐 | ★★★★ | ★★★★★ | ★★ | ★★★★ |
| 事务消息 | ✅ 原生 | 需自实现 | 弱 | ✅ |
| 顺序消息 | ✅ | ✅（按 partition） | 一般 | ✅ |
| 延迟消息 | ✅ 原生 | 需插件 | 插件 | ✅ |
| 运维复杂度 | 中 | 中 | 低 | 高 |
| 推荐 | **✅ 业务主选** | **✅ 流式主选** | 老系统兼容 | 暂不引入 |

**决策**：双轨——RocketMQ（业务）+ Kafka（流式 / AI 数据）。

## 5. AI 框架：Spring AI vs LangChain4j vs 自研

| 维度 | Spring AI | LangChain4j | 自研薄封装 |
|------|-----------|-------------|----------|
| Spring 集成 | ★★★★★ | ★★★ | ★★★★ |
| 多模型支持 | 广 | 广 | 取决投入 |
| Agent / Tool | 中（演进中） | ★★★★ | 取决投入 |
| RAG | ★★★★ | ★★★★ | 取决投入 |
| 社区活跃 | ★★★★ | ★★★★ | ★ |
| 锁定风险 | 中 | 中 | 高（自维护） |
| 推荐 | **✅ 主选** | Agent 重场景叠加 | ❌ |

**决策**：Spring AI 主选；通过 `ai-api` 抽象避免锁定；复杂 Agent 用 LangChain4j。

## 6. 向量库：PgVector vs Milvus vs Redis-Vector vs ES-kNN

| 维度 | PgVector | Milvus | Redis-Vector | ES kNN |
|------|----------|--------|--------------|--------|
| 部署复杂度 | ★（已有 PG） | ★★★ | ★（已有 Redis） | ★★ |
| 规模上限 | 千万 | 十亿 | 百万 | 亿 |
| 混合检索（关键词+向量） | ✅ | 中 | 弱 | ★★★★★ |
| 事务一致性 | ✅（与业务库同 PG） | ❌ | ❌ | ❌ |
| 推荐 | **✅ 默认** | 大规模升级 | 缓存层 | 已有 ES 时 |

**决策**：默认 PgVector；通过 `VectorStore` 抽象，规模上去切 Milvus。

## 7. 鉴权：Spring Security vs Sa-Token vs Shiro

| 维度 | Spring Security 6 | Sa-Token | Shiro |
|------|------------------|----------|-------|
| 与 Spring Boot 4 兼容 | ★★★★★ | ★★★★ | ★★★ |
| OAuth2 / OIDC / SAML | ★★★★★ | 中 | 弱 |
| 上手成本 | 高 | 低 | 中 |
| 业务侧灵活度（多端 token） | 中 | ★★★★★ | 中 |
| 推荐 | **✅ 协议层主选** | **✅ 业务侧补充（多端会话）** | ❌ |

**决策**：Spring Security 处理协议；Sa-Token 处理业务侧多端会话与权限注解。

## 8. 容器编排：Swarm vs K8s（test 环境层面）

| 维度 | Docker Swarm | K8s |
|------|-------------|-----|
| 学习成本 | 低 | 高 |
| 与生产一致性 | 中 | ★★★★★ |
| 资源占用 | 低 | 高 |
| 灰度 / HPA | 弱 | ★★★★★ |
| 推荐 | **✅ test** | **✅ beta / pro** |

**决策**：保持你的初始方案（test = Swarm，beta/pro = K8s）；如团队 K8s 熟练，可统一 K8s（考虑成本）。

## 10. 风险与缓解

| 风险 | 等级 | 缓解 |
|------|------|------|
| Spring Boot 4 GA 时间不确定 | 中 | 先按 RC 设计，BOM 抽象，GA 后小改即可 |
| 4 年期窗口过严 | 中 | 提供白名单 + 过期日机制；季度盘点 |
| 六边形对团队冲击 | 中 | archetype 双模板；提供示例与培训 |
| 多数据源动态加减引入 bug | 高 | 强 e2e 测试 + 灰度环境必跑 |
| 网关 + 灰度 + 配置总线串联复杂 | 高 | 文档 04 单独验证 PoC |
| AI 成本失控 | 高 | 配额 + 路由 + 监控（文档 03 §2.6） |
| 工具类封装引发"中间人 bug" | 中 | 充分单测 + 与原库行为对齐文档 |
