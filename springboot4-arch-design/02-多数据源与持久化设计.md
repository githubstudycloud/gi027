# 02 - 多数据源与持久化设计

> 版本：v1.0 | 日期：2026-04-27 | 状态：草稿

---

## 1. 业务模型

- **1 个公共库（common-db）**：保存 租户、业务库路由表、字典、权限、审计、配置快照。
- **N 个业务库（business-db-1 .. business-db-N）**：N ≈ 10，独立 schema，可能跨实例。
- **路由依据**：登录态中的 `tenantId` / `bizCode` → 查公共库 `t_datasource_route` → 决定本次请求落到哪个业务库。

---

## 2. 整体架构

```
                ┌────────────────────────┐
请求 ──► Filter│ TenantContextHolder    │  解析 token / header
                └─────────┬──────────────┘
                          ▼
                ┌────────────────────────┐
                │ DataSourceRouter       │  AbstractRoutingDataSource
                │  - lookupKey = bizCode │
                └─────┬──────────────┬───┘
                      │              │
              ┌───────▼─────┐  ┌────▼────────┐  ┌─────────────┐
              │ common-db   │  │ business-1  │  │ business-N  │
              │ (HikariCP)  │  │ (HikariCP)  │  │ (HikariCP)  │
              └─────────────┘  └─────────────┘  └─────────────┘
                      ▲
                      │ 启动加载 + 总线推送
              ┌───────┴───────────┐
              │ DataSourceRegistry│  动态增删业务库（无需重启）
              └───────────────────┘
```

---

## 3. 关键能力

| 能力 | 实现 |
|------|------|
| 路由 | Spring `AbstractRoutingDataSource` + `ThreadLocal` 上下文 |
| 注解切换 | `@DS("common")` / `@DS("biz")` / `@DS("biz:#{tenantId}")` SpEL |
| 动态注册业务库 | 配置中心推送 → `DynamicRoutingDataSource.add/remove` |
| 跨库事务 | **默认禁用** XA；要求业务设计为"本地事务 + 最终一致" |
| 跨库查询 | 禁止；必须落库前合并或在 application 层组装 |
| 分库分表 | 单业务库压力大时启用 ShardingSphere（可选模块） |
| 读写分离 | 通过 `@DS("biz:read")` / `@DS("biz:write")` |
| 慢 SQL | p6spy + 慢 SQL 落 `t_slow_sql` 表，超过阈值告警 |

## 4. 公共库表设计（草案）

| 表 | 用途 |
|----|------|
| `t_tenant` | 租户元数据 |
| `t_datasource` | 业务库连接元数据（JDBC URL、加密密码、池参数） |
| `t_datasource_route` | (tenantId, bizCode) → datasourceId 路由 |
| `t_datasource_change_log` | 路由变更审计 |
| `t_dict` | 全局字典 |
| `t_audit_log` | 审计日志（也可分流到 ES） |

> 密码字段统一使用 KMS 加密存储；启动时解密注入。

## 5. 切换流程（同请求内）

```
1. Web Filter 解析 tenantId → ThreadLocal
2. Service 进入业务方法
3. AOP 解析 @DS 注解 → 设置 lookupKey
4. 数据访问 → AbstractRoutingDataSource.determineCurrentLookupKey()
5. 方法返回 → AOP 还原上一层 lookupKey（栈式）
6. 请求结束 → ThreadLocal 清理
```

> 异步线程（`@Async` / 线程池 / MQ Listener）必须使用 `TtlExecutor`（`TransmittableThreadLocal`）传递上下文。

## 6. 多数据源与事务

| 场景 | 策略 |
|------|------|
| 同一业务库内 | `@Transactional` 正常工作 |
| 跨业务库 | 拆为多个本地事务；用 `transactional outbox + MQ` 实现最终一致 |
| 公共库 + 业务库 | 优先公共库只读；写场景用 outbox 模式 |
| 跨库读 | 禁止 join；改为 `application` 层聚合或冗余字段 |

## 7. 动态业务库（不停机加库）

```
配置中心 (Nacos) ──change event──► config-bus ──► DataSourceRegistry
                                                      │
                                       ┌──────────────┼──────────────┐
                                       ▼              ▼              ▼
                                   add(ds)        remove(ds)      reload(ds)
```

- 新加业务库：配置推送 → 注册 → 健康探测通过 → 进入路由表；
- 移除业务库：先标记 draining → 等待无活跃事务 → 关闭连接池 → 移出注册表；
- 全程对业务方法零感知。

## 8. 持久化框架选型

| 选项 | 优点 | 缺点 | 选用 |
|------|------|------|------|
| MyBatis-Plus | 国内生态强、SQL 可控、与 ShardingSphere 兼容好 | 抽象浅 | ✅ 主选 |
| Spring Data JPA | 抽象高、跨库语义统一 | 复杂 SQL 难写、动态多源支持弱 | ❌ |
| jOOQ | 类型安全、表达力强 | 学习曲线 | ⚠️ 备选（数仓类项目） |

## 9. 自我修订记录

| 轮次 | 修订点 |
|------|--------|
| R1 | 把"跨库事务用 XA"改为"禁用 XA + outbox"，避免性能与运维灾难 |
| R2 | 增加动态加/减库流程（不停机），呼应文档 04 的总线 |
| R3 | 异步上下文传递必须用 TTL，避免线程池里 tenantId 错乱（生产高频事故） |
