# 07 - 环境与部署拓扑设计

> 版本：v1.0 | 日期：2026-04-27 | 状态：草稿
> 对应你的需求：local（单机 Docker） / test（Swarm） / beta（K8s 灰度） / pro（K8s 生产）

---

## 1. 环境矩阵

| 环境 | profile | 编排 | 目的 | 数据 | 灰度 |
|------|---------|------|------|------|------|
| **local** | `local` | 单机 Docker / Compose | 开发自测 | 本地 / 共享开发库 | 否 |
| **test** | `test` | Docker Swarm | 集成测试、QA | 独立测试库 | 否 |
| **beta** | `beta` | K8s（与 pro 同集群或独立） | 预发 / 灰度验证 | 与 pro 镜像或独立 | ✅ 灰度 |
| **pro** | `pro` | K8s（多 AZ） | 生产 | 生产 | ✅ 蓝绿/金丝雀 |

> profile 严格对应；任何情况下**不允许混用**（如 local profile 连 pro 库）。配置中心通过命名空间隔离。

---

## 2. local（单机 Docker / Compose）

```
docker-compose.local.yml
├── app                 # 业务服务（热加载，挂载源码）
├── nacos
├── mysql / pgvector
├── redis
├── rocketmq-namesrv + broker
├── kafka + zookeeper（可选）
├── minio               # 对象存储
├── jaeger              # 链路
└── mailhog             # 本地邮件
```

- 一键启动：`make local-up`；
- 与 IDE 集成：Spring Boot DevTools + 远程 Debug；
- 资源占用大时提供 `make local-up-lite`（仅核心组件）。

---

## 3. test（Docker Swarm）

- **目的**：模拟集群行为（多节点、滚动更新、健康检查），但比 K8s 轻；
- 节点：3 manager + N worker；
- 配置：`docker stack deploy -c stack.test.yml`；
- 与 CI/CD：合并到 `develop` 分支自动部署到 test；
- 数据：独立测试库 + 测试数据集（每日凌晨重置选项）；
- 监控：Prometheus + Grafana（精简）。

> Swarm 选型理由：比 K8s 学习成本低、足够覆盖测试需求；与生产 K8s 行为接近（Service / Stack / Secret 概念近似），便于过渡。

---

## 4. beta（K8s 灰度）

- **核心定位**：流量灰度 + 真实数据 + 可回滚；
- 与生产同集群（推荐）或独立小集群；
- 部署策略：
  - **金丝雀（Canary）**：1 实例 → 5% 流量 → 25% → 50% → 100%；
  - **影子流量（Shadow）**：复制生产流量到 beta，不返回响应（用于核对）；
- 数据：原则上读生产从库 + 写沙盒（防污染），重大变更才接生产写库；
- 灰度依据：用户 ID 哈希 / 内部员工标 / 白名单租户。

## 5. pro（K8s 生产）

### 5.1 集群拓扑

```
                    ┌─────────────────────┐
                    │    Public Ingress   │ (LB + WAF)
                    └─────────┬───────────┘
                              ▼
                    ┌─────────────────────┐
                    │  Spring Cloud GW    │ (HPA, 多副本)
                    └─────────┬───────────┘
                  ┌───────────┼───────────┐
                  ▼           ▼           ▼
            [order-svc]  [user-svc]  [...]   ← Deployment + HPA + PDB
                  │
        ┌─────────┴─────────┐
        ▼                   ▼
   [common-db]        [biz-db-1..N]   ← 托管 RDS（不在 K8s 内）
                  │
        ┌─────────┴─────────┐
        ▼                   ▼
   [redis-cluster]   [rocketmq / kafka]
```

### 5.2 关键策略

| 项 | 策略 |
|----|------|
| 副本数 | 至少 2，跨 AZ 反亲和 |
| HPA | CPU + 自定义指标（QPS / 队列堆积） |
| PDB | minAvailable=1，保证发布不全挂 |
| Probe | startup / readiness / liveness 三件套 |
| 资源 | requests / limits 强制；OOMKill 监控 |
| 配置 | ConfigMap + Secret + Nacos（外部配置） |
| 镜像 | distroless / alpine；多架构（amd64 + arm64） |
| 部署 | Argo CD（GitOps），所有变更走 Git |
| 发布 | 金丝雀 + 自动回滚（基于 SLO） |
| 备份 | DB 每日全备 + binlog 增量；定期恢复演练 |

### 5.3 安全

- mTLS（Istio 可选）；
- NetworkPolicy 默认 deny，按需放行；
- Secret 由 Vault / 云 KMS 注入，不入 Git；
- 镜像扫描（Trivy）作为合并门禁；
- Pod Security Standard：restricted。

## 6. CI/CD 流水线（统一）

```
PR → Lint / Test / Build → 镜像构建 → 镜像扫描
                                          │
                       ┌──────────────────┼─────────────────┐
                       ▼                  ▼                 ▼
                   合并 develop       合并 release       合并 main
                       │                  │                 │
                   部署 test          部署 beta         部署 pro
                       │                  │           （审批 + 金丝雀）
                       └──── 自动回归 ────┘
```

- 所有环境通过同一镜像（**Build Once, Promote**），仅 profile 不同；
- 配置不在镜像里。

## 7. 自我修订记录

| 轮次 | 修订点 |
|------|--------|
| R1 | 明确 4 个环境的"目的差异"，避免 test 和 beta 职责混淆 |
| R2 | beta 引入影子流量机制，弥补金丝雀对低流量场景验证不足 |
