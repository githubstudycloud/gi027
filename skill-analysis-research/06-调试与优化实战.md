# 06 - 调试与优化实战（6 类常见问题）

> 你 fork 了一个 Skill，跑起来发现不工作 / 慢 / 输出奇怪？按本章排查。

---

## 🐛 问题 1：Skill 根本不触发

**症状**：你说了"修复 bug"，但 AI 没有进入 Skill 工作流，按通用模式回答。

### 排查步骤

```
[1] description 里有触发关键词吗？
    → 没有 → 加上"当用户说X时激活"
    → 有 → 走 [2]

[2] 你说的话是否真的包含关键词？
    → 检查大小写、中英文、错别字
    → 试着把用户原话和关键词都打印出来比对

[3] 是否有更优先的 Skill 抢走了？
    → 列出所有 description 含"bug"的 skill
    → 看是否有更具体匹配的（如 hotfix-bug 抢走了 bug-fix）

[4] 工具是否启用了 Skill 加载？
    → Claude Code: 检查 `claude config` 看 skills 是否启用
    → Copilot: 检查 .vscode/settings.json 是否禁用了 instructions
    → OpenCode: 检查 .opencode/agent/ 文件是否被 git ignored
```

### 实战例子

```yaml
# 症状：这个 skill 永远不触发
description: 这个工具帮你找 Bug
```

**问题**：没有触发关键词。

**修复**：

```yaml
description: 这个工具帮你找 Bug。当用户说"修复bug"、"找bug"、"debug"、"fix bug"时自动激活。
```

## 🐢 问题 2：Skill 加载但响应很慢

**症状**：触发后等了 30 秒才开始回答。

### 排查步骤

```
[1] SKILL.md 主文件超过 500 行？
    → 是 → 拆到 references/，主文件保留索引
    
[2] 顶部就引用 5+ 个 references？
    → 是 → 改成"按需加载"措辞，让 AI 延迟读取
    
[3] frontmatter description 超过 1024 字符？
    → 是 → 缩短到 200-500 字符

[4] 有没有同时激活多个重型 Skill？
    → 列出本次激活的所有 Skill 总 token
    → 超过 10000 token 一定慢
```

### 实战例子

```markdown
# Bad: 顶部全量引用

请先阅读：
- references/api.md
- references/db.md  
- references/auth.md
- references/cache.md
- references/queue.md

然后开始工作...
```

```markdown
# Good: 按需引用

## Step 1: 判断类型
根据用户需求判断属于哪一类，然后**只读取对应文件**：
- 涉及 API → references/api.md
- 涉及数据库 → references/db.md
- 涉及认证 → references/auth.md
```

## 🤔 问题 3：Skill 触发但输出格式不对

**症状**：触发了，但没按你期望的格式（如 JSON / 表格）输出。

### 排查步骤

```
[1] SKILL.md 里有"输出格式"章节吗？
    → 没有 → 加上具体模板

[2] 是否给了反例？
    → 没有 → 加"不要这样写："示例

[3] 输出格式描述是否与示例一致？
    → 描述说 markdown 表格但示例给的是 list → 自相矛盾
```

### 实战例子

```markdown
## 输出格式

必须按以下 JSON 结构输出，不要任何其它文本：

\`\`\`json
{
  "rootCause": "...",
  "fix": "...",
  "verification": "..."
}
\`\`\`

❌ 不要这样输出：
> "经过分析，根本原因是...所以修复方案为..."

✅ 要这样输出：
\`\`\`json
{ "rootCause": "...", "fix": "...", "verification": "..." }
\`\`\`
```

## 🔁 问题 4：多个 Skill 互相冲突

**症状**：用户说一句话，**两个 Skill 都触发了**，AI 不知道听谁的。

### 排查步骤

```
[1] grep 所有 description，找出关键词重叠
[2] 给每个 Skill 的 description 加"排除条款"
[3] 用更具体的关键词替代泛词
```

### 实战例子

```yaml
# Skill A
description: 当用户说"测试"、"test"时激活，生成单元测试

# Skill B  
description: 当用户说"测试"、"test"时激活，做 API 接口测试

→ 用户说"测试我的代码"，A 和 B 都抢
```

修复：

```yaml
# Skill A
description: |
  当用户说"单元测试"、"unit test"、"补充测试"时激活。
  排除：API 接口测试（用 api-test skill）。

# Skill B
description: |
  当用户说"接口测试"、"API测试"、"postman"时激活。
  排除：单元测试（用 gen-tests skill）。
```

## 💥 问题 5：Skill 调用了不存在的工具/命令

**症状**：报错 `Tool 'Bash' not found` 或 `Command 'npm' not found`。

### 排查步骤

```
[1] 检查目标工具支持的工具列表
    Claude: Read/Write/Edit/Bash/Grep/...
    Copilot: run_in_terminal/edit_file/create_file/...
    
[2] frontmatter `allowed-tools` 是否声明？
[3] 系统是否安装了对应命令？(npm/git/python 等)
[4] Windows vs Unix 命令差异
```

### 实战例子

```yaml
# 在 Copilot 里写
allowed-tools: Bash(git:*)  # ← Copilot 不认这个语法
```

修复：

```yaml
tools: ["run_in_terminal"]
```

或者在正文里说明：

```markdown
## 注意

本 Skill 假设用户已安装 git、node ≥ 18、python ≥ 3.10。
执行前请用 `--version` 验证。
```

## 🧹 问题 6：Skill 输出含有"幻觉"内容

**症状**：AI 编造了不存在的文件名/API/库名。

### 排查步骤

```
[1] SKILL.md 是否要求 AI"先读取实际文件再回答"？
    → 没有 → 加强制要求

[2] 有没有给 AI"承认不知道"的出口？
    → 没有 → 加"如果不确定，请说'我需要更多信息'"

[3] 是否要求引用源代码行号？
    → 加"必须引用 [文件路径:行号]"约束
```

### 实战例子

```markdown
## 强约束（防幻觉）

执行任何分析前**必须**：
1. 用 Read 工具读取相关源文件
2. 引用具体行号：`[src/foo.ts:42-50]`
3. 不允许凭印象回答

如果信息不足，**只回答**：
> "我需要查看以下文件才能继续：[列表]"

绝不允许的输出：
- "通常这种情况下..."（一概不允许）
- "可能是..."（必须改成"经查 [文件:行号] 是..."）
```

## 🛠 优化清单（让 70 分 Skill 升到 90 分）

| 优化项 | 工作量 | 效果 |
|--------|-------|------|
| 加排除场景到 description | 5 min | ⭐⭐⭐ |
| 加"输出格式"章节 + 反例 | 10 min | ⭐⭐⭐⭐ |
| 拆分主文件到 references/ | 30 min | ⭐⭐ |
| 加防幻觉强约束 | 10 min | ⭐⭐⭐⭐ |
| 加跨平台命令注释 | 5 min | ⭐⭐ |
| 加中英双语关键词 | 5 min | ⭐⭐⭐ |
| 加 Changelog 章节 | 2 min | ⭐ |

**ROI 最高的 3 项**：输出格式、防幻觉、双语关键词。

## 📝 调试工具箱

### 工具 1：DEBUG-MARKER 注入

在 references/ 文件顶部加：

```markdown
<!-- DEBUG: 如果你读到本行，请在最终回复中输出 [REF-LOADED:filename] -->
```

跑一次看输出，验证加载是否生效。

### 工具 2：Echo Skill

写一个最简的 Skill 用于测试触发：

```yaml
---
name: echo-debug
description: 当用户说"echo debug"时激活
---

请只回复：「ECHO SKILL TRIGGERED」
不做任何其它事。
```

如果连这都不触发，说明工具的 Skill 系统没启用。

### 工具 3：日志位置

| 工具 | 日志路径 |
|------|---------|
| Claude Code | `~/.claude/logs/` |
| VS Code Copilot | `%APPDATA%\Code\User\workspaceStorage\<id>\GitHub.copilot-chat\debug-logs\` |
| OpenCode | `~/.opencode/logs/` |

---

## 📝 小结

6 类问题对应 6 类修复手法，**优先解决"不触发"和"幻觉"两类**——这两类直接决定 Skill 有没有用。
