# 04 - 跨工具适配指南

> 同一个 Skill，怎么让它在 Claude / Copilot / Codex / Gemini / OpenCode / Cursor / Trae 都能跑？

---

## 🌐 主流工具的 Skill 机制对比

| 工具 | Skill 加载机制 | 文件格式 | Frontmatter 关键字段 | 子 Agent 支持 |
|------|---------------|---------|---------------------|---------------|
| **Claude Code** | `~/.claude/skills/<name>/SKILL.md` | YAML + Markdown | `name`、`description`、`allowed-tools`、`model` | ✅ Subagents |
| **VS Code Copilot** | `.github/instructions/*.md` 或扩展自带 `prompts/skills/` | YAML + Markdown | `description`、`applyTo`、`tools` | ✅ Custom Agents |
| **Codex (OpenAI)** | `~/.codex/prompts/*.md`（无原生 Skill 概念） | Markdown / JSON | `name`、`description` | ❌ 无（只能堆 prompt） |
| **Gemini CLI** | `GEMINI.md` 项目级 + `~/.gemini/extensions/` | Markdown | `description`（自由） | ⚠️ 通过 `@` 引用文件 |
| **OpenCode** | `.opencode/agent/*.md`、`.opencode/command/*.md` | YAML + Markdown | `description`、`tools`、`model`、`mode` | ✅ Agent/Subagent |
| **Cursor** | `.cursor/rules/*.mdc` | YAML + Markdown | `description`、`globs`、`alwaysApply` | ❌ 仅规则注入 |
| **Trae** | `.trae/rules/*.md` | Markdown | 无 frontmatter（接近 Cursor） | ❌ |

---

## 🔄 核心适配规则（背下来！）

### 通用兼容字段

✅ **`description`**：所有工具都认（这是为什么它最重要）
✅ **正文 Markdown**：所有工具都解析

### 工具特定字段对照

| 含义 | Claude | Copilot | OpenCode | Cursor | Codex/Gemini |
|------|--------|---------|----------|--------|--------------|
| 工具白名单 | `allowed-tools` | `tools` | `tools` | （无）| （无）|
| 文件作用域 | （无）| `applyTo: "**/*.ts"` | （无）| `globs: ["*.py"]` | （无）|
| 锁定模型 | `model` | （无）| `model` | （无）| （无）|
| 始终激活 | （无）| （无）| `mode: primary` | `alwaysApply: true` | （无）|
| 子 Agent | sub-agent file | agent mode | `mode: subagent` | （无）| （无）|

## 📦 一份 Skill，多种格式：实操示范

假设我们有一份 Claude 格式的 Skill：

```yaml
---
name: api-spec
description: API规范生成助手 - 当用户说"设计接口"、"写OpenAPI"时激活
allowed-tools: Read, Write, Bash(curl:*)
model: claude-sonnet-4
---

# API Spec Generator

## When to Use
当用户需要生成 OpenAPI 3.1 规范时...

## Workflow
1. 收集端点列表
2. 生成 YAML
...
```

### 适配成 VS Code Copilot Instruction

```yaml
---
description: API规范生成助手 - 当用户说"设计接口"、"写OpenAPI"时激活
applyTo: "**/{api,openapi,swagger}/**/*.{yaml,yml,json}"
---

# API Spec Generator
... （正文几乎不改）
```

**变更点**：
- ❌ 删 `name`（Copilot 用文件名）
- ❌ 删 `model`（不支持锁模型）
- ✅ 改 `allowed-tools` → 删除（Copilot 通过 agent 模式控制）
- ✅ 加 `applyTo`（让它只在 API 相关文件生效）

### 适配成 OpenCode Agent

```yaml
---
description: API规范生成助手 - 当用户说"设计接口"、"写OpenAPI"时激活
mode: subagent
tools:
  read: true
  write: true
  bash: false
---

# API Spec Generator
...
```

**变更点**：
- ✅ `allowed-tools` → `tools` (对象格式)
- ✅ `model` 删（或换成 OpenCode 支持的模型 ID）
- ✅ 加 `mode: subagent`

### 适配成 Cursor Rule

```yaml
---
description: API规范生成助手
globs: ["**/api/**/*.yaml", "**/openapi.yaml"]
alwaysApply: false
---

# API Spec Generator
...
```

**变更点**：
- ✅ description 缩短（Cursor 在选择器里展示）
- ✅ `applyTo` → `globs`
- ❌ 删触发关键词章节（Cursor 不靠关键词激活，靠文件匹配）

### 适配成 Codex / Gemini Prompt

Codex 和 Gemini 没有原生 Skill 体系，最简单做法：把整份 SKILL.md **当作 prompt 模板**。

`~/.codex/prompts/api-spec.md`:

```markdown
# Use this prompt when user asks to design API or write OpenAPI

You are an API spec generator. Follow this workflow:

1. 收集端点列表
2. 生成 YAML
...

User input: {{user_request}}
```

Gemini CLI 用 `GEMINI.md` 在项目根目录：

```markdown
# Project Conventions

When user mentions "API design", "OpenAPI", or "接口设计",
follow the workflow defined below:

## API Spec Workflow
... (整份正文)
```

## 🛠 自动化转换脚本思路

写一个 `convert-skill.ps1`，核心逻辑：

```powershell
param(
    [string]$Source,        # 源 SKILL.md 路径
    [string]$Target,        # claude | copilot | opencode | cursor | codex
    [string]$OutDir
)

$content = Get-Content $Source -Raw
$matches = [regex]::Match($content, '(?ms)^---\s*\n(.*?)\n---\s*\n(.*)$')
$frontmatter = $matches.Groups[1].Value
$body = $matches.Groups[2].Value

switch ($Target) {
    "copilot" {
        # 删除 name/model/allowed-tools，可能加 applyTo
        $newFm = $frontmatter -replace '(?m)^name:.*\n', '' `
                              -replace '(?m)^model:.*\n', '' `
                              -replace '(?m)^allowed-tools:.*\n', ''
    }
    "opencode" {
        # allowed-tools → tools 对象
        # 略
    }
    "cursor" {
        # description 截断到 200 字
        # applyTo → globs
    }
    # ...
}

$out = "---`n$newFm`n---`n$body"
Set-Content -Path "$OutDir\$([System.IO.Path]::GetFileName($Source))" -Value $out
```

完整版本见 `examples/convert-skill.ps1`。

## ⚠️ 常见踩坑

### 坑 1：把 Claude 的 `Bash(git:*)` 直接搬到 Copilot

Copilot 不识别这种语法，会忽略它，导致 Skill 拥有**比预期更大**的工具权限。

**解法**：迁移时改成 Copilot 的 `tools: ["run_in_terminal"]` 显式声明。

### 坑 2：把多个 Skill 合并成一个 GEMINI.md

Gemini 没有 Skill 隔离，所有内容都进系统 prompt。**超过 8000 字符就会显著拖慢响应**。

**解法**：只把最常用的 1-2 个核心 Skill 放进 GEMINI.md，其它放 `.gemini/extensions/`。

### 坑 3：触发关键词全是英文

Cursor、Trae、国内 IDE 用户输入多为中文，全英文关键词激活率极低。

**解法**：每条 description 中英双语关键词都要有。

### 坑 4：跨平台路径

Skill 里写 `bash scripts/run.sh`，到 Windows 上直接挂。

**解法**：

```markdown
## 执行脚本

- Linux/Mac: `bash scripts/run.sh`
- Windows: `pwsh scripts/run.ps1`
- 跨平台优先: `node scripts/run.js`
```

## 📊 适配难度矩阵

| 源 \ 目标 | Claude | Copilot | OpenCode | Cursor | Codex/Gemini |
|----------|--------|---------|----------|--------|--------------|
| Claude   | -      | 易      | 易       | 中     | 难           |
| Copilot  | 易     | -       | 易       | 易     | 难           |
| OpenCode | 易     | 易      | -        | 中     | 难           |
| Cursor   | 中     | 易      | 中       | -      | 中           |

**易 = 改 frontmatter 即可；中 = 还要重组章节；难 = 几乎重写**

---

## 📝 小结

1. `description` 是跨工具最大公约数，写好它等于一稿多投
2. 适配 = **重写 frontmatter** + 偶尔调整章节
3. Cursor/Trae 偏向"规则注入"，Claude/OpenCode 偏向"工作流"，Codex/Gemini 只能当 prompt 模板
4. 别把 `Bash(*)` 这种危险权限带到不识别它的工具里
5. 中英双语触发词是国内开发者必须做的本地化
