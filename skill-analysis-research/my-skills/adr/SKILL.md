# 架构决策记录（ADR）助手

ADR 用于记录重要的架构/技术决策，保存决策背景和理由，让团队未来能理解"为什么这样做"。

## 参数说明
- `$ARGUMENTS`：决策的标题或简要描述

## 工作流程

### 第一步：确定 ADR 编号

```bash
ls docs/adr/ 2>/dev/null | grep -E "^[0-9]+" | sort -n | tail -1
```

如果 `docs/adr/` 不存在则从 0001 开始；否则在最大编号 +1。

### 第二步：收集信息

如果 `$ARGUMENTS` 不够详细，询问：
1. 面临什么问题或需要做什么决策？
2. 考虑了哪些备选方案？
3. 最终选择了什么？原因是什么？
4. 有什么已知的缺点或风险？

### 第三步：生成 ADR 文件

创建文件 `docs/adr/{编号}-{标题-kebab-case}.md`：

```markdown
# ADR-{编号}: {标题}

> 日期：{今天} | 状态：草稿 → 提议 → **已批准** / 已废弃 / 已替代

## 状态
草稿

## 上下文（Context）

{描述促使这个决策的背景和问题。这里不作价值判断，只陈述事实。}

**面临的问题：**
- ...

**约束条件：**
- ...

## 决策（Decision）

{明确说明选择了什么，以及为什么。}

我们决定使用 **{选择}**，因为：
- {理由 1}
- {理由 2}

## 方案对比

| 方案 | 优点 | 缺点 | 最终 |
|------|------|------|------|
| {方案 A} | | | ✅ 选择 |
| {方案 B} | | | |
| {方案 C} | | | |

## 后果（Consequences）

**正面影响：**
- ...

**负面影响/权衡：**
- ...

**中性影响：**
- ...

## 关联
- 相关 ADR：
- 相关 Issue/PR：
- 参考资料：
```

### 第四步：更新 ADR 索引

在 `docs/adr/README.md`（不存在则创建）中追加新条目：

```markdown
| ADR-{编号} | {标题} | {今天} | 草稿 |
```
