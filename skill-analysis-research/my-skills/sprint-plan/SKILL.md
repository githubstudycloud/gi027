# 迭代规划助手

将需求拆解为可执行的开发任务，输出 Sprint Backlog。

## 参数
- `$ARGUMENTS`：PRD 文件路径或功能描述

## 工作流程

### 第一步：读取需求

如果 `$ARGUMENTS` 是文件路径，读取该文件；否则直接分析描述内容。

同时扫描项目现状：
```bash
git log --oneline -20   # 了解最近开发节奏
ls docs/                # 查看现有文档
```

### 第二步：层次化拆解

按三层结构拆解：

```
Epic（史诗）：大功能模块，2-4 周
  └── Story（用户故事）：可独立交付的价值单元，1-3 天
        └── Task（开发任务）：具体的技术任务，< 1 天
```

**拆解原则：**
- Story 必须独立可测试、可演示
- Task 粒度：一个人一天内可完成
- 每个 Story 包含：前端 + 后端 + 测试（不要割裂）
- 优先考虑：核心路径 → 边界场景 → 优化

### 第三步：估算工作量（故事点）

```
1 点  = 半天（简单改动，逻辑清晰）
2 点  = 1 天（标准 CRUD，有一定复杂度）
3 点  = 1.5-2 天（中等复杂，跨层交互）
5 点  = 3 天（复杂功能，需要设计讨论）
8 点  = 应拆分（太复杂，必须拆小）
```

### 第四步：输出迭代计划

