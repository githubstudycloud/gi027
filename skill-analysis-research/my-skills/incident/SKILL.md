# 故障响应助手

**按 MTTR（平均恢复时间）最优路径处理生产故障。**

## 参数
- `$ARGUMENTS`：故障描述、报错信息或告警内容

## 故障响应流程（OODA Loop）

### 🚨 阶段 0：立即响应（0-5 分钟）

**第一问：影响了多少用户/功能？**

```
影响程度快速判断：
P0（全站故障）: 主要功能不可用，影响 > 50% 用户 → 立即升级
P1（核心功能故障）: 核心流程受损，影响 > 10% 用户 → 30分钟内响应
P2（部分功能故障）: 非核心功能，< 10% 用户 → 2小时内响应
P3（性能问题）: 功能可用但慢 → 正常工作时间处理
```

**立即询问：**
- 什么时间开始出现问题？
- 最后一次部署是什么时间？
- 是否有告警触发（监控/用户反馈）？

### 🔍 阶段 1：快速定位（5-15 分钟）

执行系统诊断：

```bash
# 1. 检查服务状态
docker ps -a 2>/dev/null || kubectl get pods -A 2>/dev/null

# 2. 查看最近日志（最后 100 行）
docker logs --tail=100 {service-name} 2>/dev/null

# 3. 检查错误率（如果有 metrics）
# curl http://localhost:9090/api/v1/query?query=rate(http_requests_total{status=~"5.."}[5m])

# 4. 检查数据库连接
# psql $DATABASE_URL -c "SELECT count(*) FROM pg_stat_activity;"

# 5. 检查内存/CPU
# kubectl top pods 2>/dev/null || docker stats --no-stream
```

**根据 `$ARGUMENTS` 中的错误信息，定向搜索：**

```bash
# 搜索相关代码
grep -rn "{关键错误词}" src/ --include="*.java" --include="*.py" -l

# 检查最近的 git 变更（回归点）
git log --oneline --since="24 hours ago"
```

### 🛡️ 阶段 2：止损（15-30 分钟）

**按情况选择止损方案：**

| 情况 | 止损方案 | 命令 |
|------|---------|------|
| 刚部署引起 | 立即回滚 | `git revert` 或触发回滚流水线 |
| 数据库迁移问题 | 执行 downgrade | `alembic downgrade -1` |
| 第三方服务故障 | 启用降级/熔断 | 切换降级开关 |
| 流量峰值 | 临时限流 | 调整网关限流配置 |
| 内存泄漏 | 重启服务 | 滚动重启（最后手段） |

**⚠️ 重要：止损优先于根因分析，先恢复服务，再排查原因。**

### 📊 阶段 3：生成故障时间线

```markdown
## 故障时间线

| 时间 | 事件 |
|------|------|
| HH:MM | 告警触发 / 用户反馈 |
| HH:MM | 开始调查 |
| HH:MM | 确认影响范围 |
| HH:MM | 发现根因 |
| HH:MM | 执行止损措施 |
| HH:MM | 服务恢复 |
| HH:MM | 用户通知 |

**MTTR（平均恢复时间）：{N} 分钟**
```

### 🔎 阶段 4：根因分析（恢复后）

使用 **5 Why 分析法**：

```
问题：{描述症状}

Why 1: 为什么服务不可用？
  → {直接原因}

Why 2: 为什么 {直接原因}？
  → {更深一层}

Why 3: 为什么 {上一层原因}？
  → {更深层}

Why 4: 为什么 {上一层原因}？
  → {系统/流程原因}

Why 5: 为什么 {上一层原因}？
  → 根本原因：{流程/架构/团队层面的问题}
```

### 📝 阶段 5：输出故障报告（RCA）

```markdown
# 故障复盘报告（Post-Mortem）

> 时间：{故障时间} | 级别：P{N} | 持续：{N} 分钟
> 状态：已解决 | 撰写人：- | 日期：{今天}

## 影响
- 受影响功能：{列表}
- 受影响用户：约 {N}（{%}）
- 业务损失：{如有}

## 时间线
{故障时间线表格}

## 根本原因
{5 Why 分析结论}

## 解决方案
{执行了什么止损/修复措施}

## 后续改进（Action Items）

| 改进项 | 类型 | 优先级 | 负责人 | 截止 |
|--------|------|--------|--------|------|
| 添加 {指标} 告警 | 监控 | P0 | - | 本周 |
| 补充 {场景} 测试 | 测试 | P1 | - | 下个Sprint |
| 优化 {组件} 的 {问题} | 技术债 | P1 | - | 下个Sprint |

## 经验总结
{团队可以学到什么，如何避免同类故障}
```

**最后：将高优先级 Action Items 添加到 /tech-debt 和下期 Sprint。**
