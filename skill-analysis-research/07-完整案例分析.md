你想：

## 步骤 1：5 步阅读法（来自 [01-如何阅读别人的Skill.md](./01-如何阅读别人的Skill.md)）

### ① 目录树

```powershell
PS> Get-ChildItem -Path "c:\Users\John\.claude\skills\bug-fix" -Recurse
```

输出（设想）：

```
bug-fix/
└── SKILL.md
```

→ **类型：单文件简单工作流型** ⭐

### ② Frontmatter

```yaml
---
name: bug-fix
description: Bug分析修复助手 - 深度分析Bug根因，定位代码，给出修复方案。
             当用户说"修复bug"、"排查问题"、"报错了"、"fix"时自动激活。
argument-hint: "[错误信息 或 Bug描述]"
allowed-tools: Read, Write, Edit, Glob, Grep, Bash
context: fork
agent: bug-analyzer
---
```

**字段诊断**：

| 字段 | 评价 |
|------|------|
| `name` | ✅ 合规 |
| `description` | ✅ 中英文触发词、有作用说明 |
| `argument-hint` | ✅ 给出参数提示（Claude Code 原生支持）|
| `allowed-tools` | ⚠️ `Bash` 没有限定子命令，权限偏大 |
| `context: fork` | ⚠️ Claude Code 特有字段，迁移到 Copilot 会丢失 |
| `agent: bug-analyzer` | ⚠️ 引用了 sub-agent，迁移时需注意 |

### ③ 章节骨架

```powershell
PS> Select-String -Path "c:\Users\John\.claude\skills\bug-fix\SKILL.md" -Pattern "^#{1,3} "
```

预期发现：

```

### 阶段 1：信息收集

### 阶段 3：根因分析
### 阶段 4：修复实施
### 阶段 5：回归验证
```

→ **结构清晰，5 阶段流程** ✅

### ④ 引用关系

```powershell
PS> Select-String -Path "c:\Users\John\.claude\skills\bug-fix\SKILL.md" -Pattern "\[.*\]\(.*\)|references/"
```

预期：无外部引用，纯单文件。

→ **无依赖图谱**，简单。

## 步骤 3：让 AI 帮你解读不懂的部分（来自 [05-借助AI理解Skill.md](./05-借助AI理解Skill.md)）

打开 ChatGPT/Claude，用 **Prompt 6（定向追问）**：

```
我看不懂 SKILL.md frontmatter 里 `context: fork` 是什么意思，
也不知道 `agent: bug-analyzer` 怎么生效。

【片段】
context: fork
agent: bug-analyzer

请回答：
1. 这两个字段在 Claude Code 里干什么？
2. 迁移到 VS Code Copilot 时怎么处理？
3. 不要这两个字段最坏会发生什么？
```

AI 回答（典型）：

> 1. `context: fork` 表示这个 skill 在隔离的子上下文里运行，不污染主对话。`agent: bug-analyzer` 是声明用一个名为 `bug-analyzer` 的子 agent 执行（需要该 agent 已存在）。
> 2. Copilot 没有等价字段。可以通过 `chatmode: agent` 或者直接通过 Custom Agent 文件实现类似隔离。
> 3. 删掉这两个字段后，Skill 仍然可用，只是会污染主对话上下文。如果不在乎上下文隔离，可以接受。

## 步骤 4：跨工具适配（来自 [04-跨工具适配指南.md](./04-跨工具适配指南.md)）

### 适配为 VS Code Copilot 格式

新建 `.github/instructions/bug-fix.instructions.md`：

```yaml
---
description: Bug分析修复助手 - 深度分析Bug根因，定位代码，给出修复方案。当用户说"修复bug"、"排查问题"、"报错了"、"fix"时自动激活。
applyTo: "**"
---

# Bug 分析修复助手

> 改编自 Claude Code skills/bug-fix（原作者：本仓库 .claude/skills/bug-fix/SKILL.md）
> 改造：删除 `context: fork`、`agent: bug-analyzer`（Copilot 无等价字段）
> 改造：删除 `argument-hint`（Copilot 通过对话获取参数）

## 工作流程

### 阶段 1：信息收集
（保持原样）

### 阶段 2：复现验证
（保持原样）

...
```

**适配要点**：
- ❌ 删除 `name`（Copilot 用文件名）
- ❌ 删除 `argument-hint`（Copilot 不支持）
- ❌ 删除 `allowed-tools`（改用 `tools` 或省略）
- ❌ 删除 `context: fork` 和 `agent: bug-analyzer`
- ✅ 增加 `applyTo: "**"` 表示全局生效

## 步骤 5：优化（来自 [06-调试与优化实战.md](./06-调试与优化实战.md)）

针对评分扣分项，做 4 处优化：

### 优化 1：加排除场景（B5，+6 分）

```yaml
description: |
  Bug分析修复助手 - 深度分析Bug根因，定位代码，给出修复方案。
  
  触发：当用户说"修复bug"、"排查问题"、"报错了"、"fix"、"debug"时激活。
  
  不适用：
  - 用户只想理解代码（用 explain-code skill）
  - 用户要新建功能（用 new-feature skill）
  - 性能优化（用 perf-tune skill）
```

### 优化 2：限定 Bash 子命令（E1，+3 分）

```yaml
allowed-tools: Read, Write, Edit, Glob, Grep, Bash(git:*), Bash(npm test:*), Bash(pytest:*)
```

只允许 git 查询、跑测试，不允许任意 shell。

### 优化 3：加输出格式约束 + 防幻觉（C5+幻觉防御）

在正文末尾加：

```markdown
## 输出格式（强约束）

最终回复必须包含 4 个固定章节：

### 1. 根因（Root Cause）
- 必须引用具体文件行号：`[src/foo.ts:42]`
- 不允许"可能是…"，必须明确

### 2. 修复方案（Fix）
- 给出 diff 格式

### 3. 验证步骤（Verification）
- 列出运行的测试命令

### 4. 风险评估（Risk）
- 这个修复可能影响哪些其它代码

❌ 禁止输出：
- "通常这种情况是…"
- "可能由…引起"
- 任何未引用源代码行号的断言
```

### 优化 4：加 Changelog（F3，+3 分）

```markdown
## Changelog

- 2026-04-28：本地化，加排除场景、限定 Bash 权限、加防幻觉约束
- 原版来源：~/.claude/skills/bug-fix/SKILL.md
```

## 🎯 这个案例的关键 takeaway

1. **真实生产 Skill 也只有 70-80 分**，别神话别人的代码
2. **30 分钟优化空间至少 +15 分**，永远值得本地化
3. **跨工具迁移 ≠ 简单复制**，要逐字段映射
4. **优化的 80% 价值在 description（B5）和输出格式（防幻觉）两处**
